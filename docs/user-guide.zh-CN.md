# Kaswin 用户指南（当前候选状态）

## 重要提示

当前工作树中的 `raffle-vnext-liveness-guard-b1000` 是本地集成候选协议，并非 Testnet 10 或 Mainnet 发布版本。请**不要**用真钱或常用钱包尝试创建、购买、开奖或退款 vNext 轮次；真实网络与钱包 E2E 尚未完成，独立安全审计也尚未完成。

可以安全进行的操作是本地构建与验证：

```powershell
npm run compile:vnext
npm run verify
npm run dev
```

当前协议为 `raffle-vnext-liveness-guard-b1000`：退款网络费从选中的购票款中扣除；首次退款还会从首个选中批次扣除启动退款的实际网络费。未到期且未达到最低票数的轮次可由任何用户主动补充 carrier，但补充者不会因此获得票权、控制权或资金提取权。

本地页面、Indexer、artifact 和单文件构建通过并不表示节点会接受交易，也不表示可公开发布。需要的 Testnet A–E、Mainnet 小额、KasWare/Kastle、移动端和审计证据列在 [开发验证循环](development-verification-loop.md) 与 [验证证据矩阵](audit-evidence-matrix.md)。

## 钱包与安全边界

日后执行真实网络验收时，必须使用独立的 Creator、Buyer A/B/C、Outsider、Sponsor 和 Mainnet 小额钱包，Testnet 与 Mainnet 严格隔离。页面只应请求钱包签名；绝不输入助记词或私钥。任何钱包签名前都要人工核对网络、账户、票数、票价、carrier、手续费及全部输出。

连接错误网络、未同步节点、错误地址或 stale UTXO 时，资金操作必须停止。stale 购买只能刷新最新 covenant cursor 后由用户重新审阅并重新签名，不能后台静默重签。`Rejected transaction` 是节点拒绝某个候选交易的总类信息，不可能从网络层彻底消失：页面会继续显示节点原始原因和本地计算交易 ID；父交易未传播会等待父交易可见，并发旧输入会只读刷新轮次列表，重复/已知交易会先核验该 ID。任何情况都不会自动弹出替代签名。

创建新轮次时预计出现 2 次钱包请求：1 次创建 covenant、1 次发布 Registry；购票只出现 1 次钱包请求，票款和 successor 位于同一笔交易。Mainnet 与 Testnet 默认 Registry 的净费用都为 **0.01 KAS**，钱包网络费另计。为满足标准中继，页面先发送临时 0.20 KAS marker，再自动返回 0.19 KAS；直接创建 0.01 KAS 输出会超过当前标准 storage mass，因此不能用作等价实现。

如果 Registry marker 已发布、但自动返回 0.19 KAS 的步骤中断，当前轮次会显示单独的恢复入口。恢复会先查询 marker 的链上花费，验证固定收款人确实是该轮创建者；无法确认状态时不会盲目重复提交，也不会把当前连接钱包改成收款人。

## 历史轮次

`raffle-v16-dynamic-refund-transition`、v15 和 v14 是历史协议，不是本地 vNext 候选的操作入口。已知历史轮次应显示对应 release 的兼容指引；未知协议只读展示。详见 [合约兼容性](contract-compatibility.md)。

## 当前 vNext 限制

vNext 退款从本次选中的购票款中扣除实际网络费，不再要求创建者按潜在买家数量预存退款费。为保证最小购买也能覆盖最坏退款费用，当前最低票价为 **1 KAS**；启动退款费用上限和单笔退款费用上限均为 0.20 KAS，因此最坏单票退款仍至少得到 0.60 KAS。单笔上限仍超过当前实测最坏质量费的两倍，并阻止公开触发者把旧 0.60 KAS 上限作为不必要矿工费烧掉。创建时可配置 `max_batches`，默认值为 100、协议硬上限为 1000。页面按 `min(1000, floor(售票秒数 / 6))`（最小 1）实时给出建议值；超过建议值仍可创建，但会提示单 Round UTXO 串行购票带来的 stale 冲突风险。不要把旧版“13 批退款”、旧 carrier 数字或旧 Registry 费用当作当前协议的费用承诺。

vNext 默认 carrier 为 **0.573 KAS**，用于保证当前费用上限内的公开开奖与关闭交易保留标准输出。即使轮次不是由本页面创建，合约和页面也会在每次购票前重新检查 carrier；低于 0.573 KAS 时禁止购票，必须先补足。只有在**未到期且尚未达到最低票数**时可以补充；补充操作需单独确认和签名，且不会获得票权或改变票号。达到最低票数后补充会移动开奖 DAA，因此合约会拒绝，当前新轮次也不需要靠补充才能完成费用上限内的开奖。旧 vNext 候选不能原地升级，必须按兼容性页面处理。

“销售截止”会让页面停止打开新的购票签名请求并开放开奖或退款。Kaspa 链上没有 upper timelock，所以截止前已存在的 covenant UTXO 理论上仍可能有一笔在途交易竞争；合约保证它最多产生一个 successor，并把该 successor 的开奖随机区块移到确认后的未来。请把截止理解为结算开放点，而不是声称完全不存在链上竞态的中心化关闸。

若到期时一张票也未售出，页面会自动切到开奖/结算页并只显示 **关闭空轮次**。这种轮次没有买家和购票款，所以不能执行普通退票；链上到期且 `soldTickets = 0`、`soldBatches = 0` 后，任何在线用户都可以代为触发，不要求创建者钱包在线或签名。合约只允许把 `carrier - 实际关闭网络费` 支付到创建者公钥，不会创建 successor covenant，也不能改收款人。

“当前抽奖”和“购买抽奖票 / 开奖并派奖”区域在未选择轮次时也会始终显示为空状态。需要选择、换用或创建轮次时，点击当前抽奖卡片右上角的 **查找 / 创建**；管理面板会在当前抽奖下方展开，再次点击即可收起，不会替换当前轮次操作区。
